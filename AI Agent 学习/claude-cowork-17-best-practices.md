## 背景

作者从 Claude Cowork 上线第一天（1月12日）开始使用，7周内运行了 400+ 次会话，测试了所有插件、连接器和斜杠命令。

**核心洞察：** 效果差距与提示词技巧无关，关键在于**设置、结构和这17个具体实践**。

> ChatGPT 时代奖励提示词工程，Cowork 时代奖励**系统工程**。

## Part 1：上下文架构（实践 1–5）

### 1. 为每个工作文件夹建立 `_MANIFEST.md`

**影响最高、最少人知道的实践。**

**问题：** Cowork 会读取文件夹里的所有内容，包括过期的旧版本文件，导致矛盾的输出。

**解决方案：** 创建 `_MANIFEST.md`，告诉 Claude 哪些是真实来源，哪些目录对应哪些领域，哪些要跳过。

三层结构：
- **Tier 1（权威）**：Claude 必须优先阅读的真实来源文件（品牌指南、项目简报、当前策略）
- **Tier 2（领域）**：按主题映射的子文件夹，只在任务涉及该领域时加载
- **Tier 3（归档）**：旧草稿、已废弃版本，Claude 忽略，除非显式请求

> 下划线前缀让它排在文件夹顶部。5分钟填写，节省数小时混乱输出。

### 2. 用全局指令作为永久操作系统

**Settings → Cowork → Edit Global Instructions**

全局指令在一切之前加载——在文件、提示词、文件夹之前。

作者的全局指令示例：
```
我是[姓名]，[角色]。开始任何任务前，先找 _MANIFEST.md 并优先读 Tier 1 文件。
执行前总是提问澄清。行动前展示简短计划。默认输出格式：.docx。
不用填充语言，不要填充输出。质量标准：每个交付物应该无需编辑即可直接给客户。
如果信心不足，明确说出来。
```

### 3. 创建三个持久上下文文件

在 "00_Context" 文件夹中创建：

| 文件 | 内容 |
|------|------|
| `about-me.md` | 你的职业身份、服务对象、当前优先级、最佳作品示例 |
| `brand-voice.md` | 沟通风格、常用词、禁用词、格式偏好、实际写作样本 |
| `working-style.md` | Claude 的协作规则、输出格式默认值、质量标准、需要避免的事项 |

**关键洞察：** 这三个文件会**复利增长**。每次 Claude 输出不满意时，问自己是提示词问题还是上下文问题。九次中有八次是上下文问题，在对应文件加一行，永久修复。

### 4. 用文件夹指令添加项目专属上下文

- **全局指令**：每次会话通用
- **文件夹指令**：特定于当前工作文件夹（客户名称、项目目标、专业术语、交付格式）

三层叠加：全局指令（通用行为）→ 文件夹指令（项目上下文）→ 提示词（具体任务）

### 5. 永远不要让 Claude 读取所有内容——主动限定范围

更大的上下文 ≠ 更好的输出。无关文件越多，噪音越多，输出质量越差。

在全局指令中加入：
```
开始任何任务时，先找 _MANIFEST.md。加载 Tier 1 文件。
只有任务明确涉及该领域时才加载 Tier 2 文件。除非我特别要求，否则不加载 Tier 3 文件。
```

## Part 2：任务设计（实践 6–10）

### 6. 定义终态，而非过程

Cowork 不是聊天机器人，是同事。不要告诉他怎么做，告诉他"完成"是什么样子。

**差的提示词：** "帮我整理文件。"

**好的提示词：** "将这个文件夹里所有文件按客户名称整理到子文件夹。所有文件名使用 YYYY-MM-DD-描述性名称格式。创建记录所有更改的摘要日志。不要删除任何内容。如果文件可能属于多个客户，放入 /needs-review。"

每个任务提示词应该回答三个问题：
1. "完成"是什么样子？
2. 限制条件是什么？
3. 遇到不确定情况 Claude 应该怎么做？

### 7. 执行前总是请求计划

在全局指令中加入：
```
对任何任务，行动前展示简短计划。等待我批准后再执行。
```

**成本：** 每次任务多 30 秒。**收益：** 永远不需要撤销一个 20 分钟的自主错误。

### 8. 告诉 Claude 如何处理不确定性

大多数人给了清晰的指令，但没说边缘情况怎么处理。Claude 会猜测，而猜测往往是错的。

在每个任务中加入不确定性处理：
```
如果日期不清楚，标记为 VERIFY。
如果文件可以放入多个文件夹，放入 /needs-review。
如果分类置信度低于 80%，标记而不是猜测。
```

### 9. 将相关工作批量放入单次会话

每次会话都有启动成本。五个相关任务合并成一次会话，不仅更快更便宜，而且质量更高（前一个任务的上下文会传递到下一个）。

### 10. 主动使用子 Agent 并行处理

触发方式：在提示词中加入 "Spin up subagents to..." 或 "Work on these in parallel using subagents"。

示例：
```
我在评估四家供应商。启动子 Agent 分别研究每家的定价、支持口碑和集成选项，给我一个对比表。
```
顺序研究需要 40 分钟，并行处理只需 10 分钟。

## Part 3：自动化与调度（实践 11–13）

### 11. 用 `/schedule` 调度重复任务

作者的调度任务示例：
- **周一早报**：每周一 7:00 汇总 Slack 频道和本周日历，保存到 /weekly-briefings
- **周五状态报告**：每周五 16:00 从 Asana 提取已完成任务，草拟状态更新
- **每日竞品跟踪**：每天 9:00 研究竞品新动态，只有新信息时才保存摘要

⚠️ 限制：调度任务只在电脑开机且 Claude Desktop 打开时运行。

### 12. 一次构建，每周运行——将一切外部化到文件

Cowork 的会话间没有记忆，这既是限制也是设计特性（无上下文污染）。

**解决方案：** 将一切外部化到文件。偏好在上下文文件中，工作流程在技能文件中，决策在日志文件中。

### 13. `/schedule` + 连接器组合实现真正的自动化

连接 Gmail、Slack、Google Drive、Notion、Asana 等 50+ 集成，然后调度任务拉取实时数据。

从 Slack 和 Gmail 开始，这两个每周能节省数小时。

## Part 4：插件与技能（实践 14–16）

### 14. 组合多个插件叠加能力

插件是可组合的，可以在单个任务中使用多个插件的能力。

作者的插件组合：
- Productivity（常驻）
- Data Analysis（常驻）
- Sales / Marketing（按工作重心轮换）

### 15. 为你的特定工作流构建自定义技能

技能文件结构：
```markdown
# [技能名称]
## Purpose: 这个技能做什么
## Inputs: Claude 需要什么信息
## Process: 步骤说明
## Output: 交付物是什么样子
## Constraints: 规则和护栏
```

保存为 `.md` 文件放入工作文件夹，或通过 Customize 菜单上传。

### 16. 用 Plugin Management 插件对话式构建插件

安装 Plugin Management 插件，说 "Help me create a plugin for [your workflow]"，Claude 引导你通过对话定义技能、斜杠命令和配置——无需代码，无需学习 Markdown 语法。

对于团队：2 月 Anthropic 推出了私有插件市场，管理员可以创建并分发自定义插件给整个组织。

## Part 5：安全与效率（实践 17）

### 17. 把 Cowork 当有能力的员工，而不是玩具

Cowork 有真实的文件系统访问权限，可以创建、移动、重命名甚至删除文件，可以浏览网页，可以与连接的工具交互。

**不可协商的安全实践：**
- ✅ 实验前先备份（尤其是文件整理任务）
- ✅ 将敏感文件放在 Cowork 从不接触的独立文件夹
- ✅ 除非特意想要删除，总是加上 "不要删除任何内容"
- ✅ 新工作流的前几次运行要监控
- ⚠️ 注意提示词注入风险：不要指向不信任的文件来源或陌生 URL
- ✅ 跟踪使用量：批量相关工作，使用文件预加载上下文而非重复解释

## 总结：17个实践背后的模式

**投入设置，减少提示词。**

| 状态 | 工作方式 |
|------|---------|
| 挣扎中的用户 | 每次任务写详细长提示词，输出结果不稳定 |
| 高效用户 | 花一个下午构建上下文架构（Manifest、全局指令、上下文文件、自定义技能），现在用10字提示词产出客户级交付物 |
